주식 자동매매(반자동 알림형) 데이터 수집 계획안 초안 v0.1
1. 문서 목적
본 문서는 다음 목적을 가진다.
자동매매 프로그램 개발 이전 단계에서 필요한 데이터 수집 및 구조화 전략을 설계한다.
초기 대상 시장(한국주식), 투자 주기(스윙/중기), 운영 방식(신호 생성 + 알림 + 수동 주문)에 맞는 현실적이고 확장 가능한 수집 아키텍처를 정의한다.
향후 미국주식, 데이트레이딩/초단타, 실시간성 고도화로 확장할 수 있도록 단계적 설계 기준을 제시한다.
2. 사전 조건 및 운영 범위 (사용자 제공 조건 반영)
2.1 대상 시장
1차: 한국주식 (KOSPI / KOSDAQ 중심)
2차 확장: 미국주식 (NYSE / NASDAQ 등)
2.2 투자 주기
초기:스윙 / 중기 (수일 ~ 수주)
확장:데이트레이딩 → 초단타
2.3 전략 스타일
멀티팩터 + 텍스트 보조형
정형 데이터: 가격/거래량/기술지표/매크로 지표/공시 메타
비정형 데이터: 뉴스/공시 원문/유명인 트윗/기업분석자료
LLM 기반 텍스트 구조화 및 해석 보조
2.4 실시간성 요구
초기 단계는 스윙/중기 중심이므로 수분 내 대응 가능
즉시 체결/초저지연 불필요
대신 데이터 정확성, 구조화 품질, 해석 일관성 우선
2.5 자동매매 범위
시스템이 직접 주문하지 않음
신호 생성 → 카카오톡/문자/텔레그램 등 알림 → 사람이 직접 주문
따라서 설계 우선순위:
신뢰도 높은 근거 제공
알림 시점/근거/리스크 설명 가능성
오탐/중복 알림 최소화
2.6 LLM 사용 조건
Gemini 2.5 Pro (ADC 인증 방식) 사용 예정
주요 역할:
뉴스/공시/리포트 요약
이벤트 분류
감성/영향도 점수 추정 (보조)
엔터티/숫자 추출
설명 가능한 알림 문구 생성
3. 목표 시스템 개요 (초기 버전)
3.1 핵심 사용자 시나리오 (사용자 제공 시나리오 반영)
사용자가 특정 종목명 또는 티커를 입력하면, 해당 종목 기준 실행일자 시점에서 1년치 데이터 수집 파이프라인이 동작한다.수집 범위는 가격정보, 뉴스, SNS, 공시 등이며, 주요 경제지표는 일단위로 사전 수집/DB화되어 있고, 이후 매매 판단에 함께 활용된다.

LLM/NLP 파이프라인으로 텍스트 구조화
피처/시그널 생성

사용자 채널(텔레그램/문자/카카오)로 전달
4. 설계 원칙 (전략적 기준)
4.1 “예측 모델”보다 “데이터 신뢰성” 우선
초기 성능은 모델보다 데이터 품질에 더 크게 좌우된다.
정확한 시점
중복 제거
이벤트 정규화
종목 매핑 정확도
공시/뉴스 원문 보존
4.2 비정형 데이터는 반드시 정형화하여 저장
LLM 자유문장 출력만 저장 금지.반드시 JSON Schema 기반 구조화 필드 저장.
4.3 이벤트 중심 구조(Event-Driven Data Model)
가격 시계열 중심만으로는 텍스트/매크로/공시 결합이 어렵다.
event 중심 레이어를 두고, 각 source를 연결
4.4 시간축 일관성 (매우 중요)
저장은 UTC 기준 권장, 표시만 KST
event_time /publish_time /ingest_time 분리
장전/장중/장후 플래그 유지
4.5 재현성 확보
백테스트/검증을 위해 다음 버전 정보 저장:
모델 버전
프롬프트 버전
전처리 파이프라인 버전
스코어링 로직 버전
4.6 단계별 확장 가능성
초기 스윙/중기 설계가 이후 데이트레이딩/미국주식 확장 시 재사용 가능해야 함.
5. 전체 데이터 자산 설계 (Data Inventory Blueprint)
아래는 수집 대상 데이터의 카테고리/목적/우선순위를 정의한 인벤토리 초안이다.

7. 데이터 수집 파이프라인 설계 (초기 버전)
아래는 초기 운영 목적(반자동 알림형)에 맞춘 두 가지 파이프라인 구조다.
7.1 파이프라인 A: 사전 적재형 (Daily/Background Ingestion)
목적
종목 입력과 무관하게 지속 수집이 필요한 공통 데이터(매크로/시장기본/캘린더)를 DB에 축적.
대상
거래 캘린더
종목 마스터
주요 경제지표 캘린더/발표값
시장 지수/환율/금리 시계열
주요 뉴스/공시/SNS (관심 universe 기준 사전 수집 가능 시)
실행 주기
일간 배치 + 이벤트/수분 단위 polling 혼합
처리 단계
수집
원본 저장(raw)
정규화(parsed)
품질 검사(QC)
LLM 구조화(선택적 비동기)
피처 테이블 반영
설계 포인트
종목 단건 요청 전에 이미 축적된 데이터가 많을수록 응답/분석 속도 향상
경제지표는 반드시 사전 적재형으로 운영 (사용자 시나리오 반영)
7.2 파이프라인 B: 종목 요청 기반 1년치 수집 파이프라인 (On-demand Ticker Backfill)
목적
사용자가 종목을 입력했을 때 해당 종목 기준으로 1년치 분석용 데이터를 빠르게 구성.
입력
ticker_or_name
as_of_date (기본값: 실행일)
lookback window = 365d
분석 모드 (quick / full) [선택]
출력
종목 분석용 데이터셋 (정형 + 텍스트 구조화 피처)
시그널 스코어
알림 후보 레코드
단계별 설계
Step 1. 종목 식별
입력값 정규화 (공백/특수문자/대소문자)
종목명→티커 매핑
동명이인/유사명 후보 처리 규칙
종목 마스터 검증
Step 2. 기간 계산
as_of_date 기준 lookback 365일
거래일 기준 window로 변환
휴장일 포함 시 캘린더로 보정
Step 3. 가격 데이터 수집/병합
종목 일봉 1년치
시장지수/업종지수 동기간
필요 시 장중 최신값 (분석 시점 업데이트)
Step 4. 공시 수집
기간 내 공시 메타/원문 회수
정정공시 체인 연결
LLM 구조화 큐 등록
Step 5. 뉴스 수집
기간 내 종목 관련 뉴스 회수
유사 기사 클러스터링
LLM 구조화 큐 등록
Step 6. SNS/트윗 수집 (선택/옵션)
종목 및 시장 관련 키워드/계정 기반 회수
신뢰도 필터 적용
LLM 구조화 큐 등록
Step 7. 기업분석자료 수집 (가능 범위)
종목 관련 문서 1년치
텍스트 추출 및 LLM 구조화
Step 8. 매크로 데이터 merge
사전 적재된 지표 DB에서 해당 기간 데이터 join
발표 이벤트와 가격 반응 구간 align
Step 9. 피처 생성
가격/기술지표
이벤트 기반 피처
텍스트 기반 점수
리스크 피처
Step 10. 신호 평가 / 알림 후보 생성
룰 기반 + 점수 기반
알림 임계값 충족 시 후보 생성
중복 알림 방지 체크
Step 11. 요약/설명 생성 (Gemini)
매수/관망/주의 후보 판단 요약
근거/반대근거/체크포인트/리스크
사람이 주문하기 위한 설명 문구 생성

10. 시간/이벤트 모델링 규칙 (중요 설계 규칙)

21. 파이프라인 시퀀스 설계서 (실행 흐름 상세)
목적: 실제 구현 시 작업 큐/배치/재시도 정책까지 고려한 실행 순서 정의
21.1 파이프라인 분류
A. 사전 적재형 파이프라인 (Background)
P1: 종목 마스터 동기화
P2: 거래 캘린더 동기화
P3: 매크로 캘린더/발표값 수집
P4: 뉴스 수집 (관심 universe 사전 수집 여부 선택)
P5: 공시 수집
P6: SNS 수집 (선정 계정)
P7: LLM 구조화 큐 소비 (비동기)
P8: 피처 재계산 배치
P9: 알림 발송 워커
B. 종목 요청형 파이프라인 (On-demand Backfill)
T1: 입력 종목 식별
T2: 1년 데이터 회수 및 병합
T3: 텍스트 구조화/피처 생성
T4: 신호 평가 및 알림 후보 생성
21.2 On-demand 종목 1년치 수집 파이프라인 상세 시퀀스
입력
ticker_or_name
as_of_date (기본값: 오늘)
lookback_days (기본 365)
mode (quick / full)
quick: 기존 DB 우선 조회 + 부족분만 수집
full: 지정 기간 재수집/재정규화 (감사/재검증용)
Step T1. 요청 접수 및 파라미터 표준화
입력값 공백/특수문자 정리
종목명/티커 판별
as_of_date를 KST 기준으로 확정 후 UTC 변환
기간 산출:
캘린더 365일 기준
내부적으로 거래일 기준 윈도우도 병행 계산
실패 처리
종목 미식별 → 후보 목록 반환 (상위 N개)
상장폐지/비활성 종목 → 분석 가능 여부 정책 적용
Step T2. 종목 식별/매핑
instrument_master exact match (ticker)
instrument_alias exact match (alias_name)
유사도 match 후보 생성
최종 instrument_id 확정
로그 필수
match_method
match_confidence
후보군 목록 (디버깅용)
Step T3. 가격 데이터 확보
price_daily 기간 조회
누락 구간 검사
누락 시 소스 재조회 → upsert
시장/업종/환율 보조 시계열 확보
일봉 품질검사 수행
품질 규칙 예시
고가 < 저가면 오류
음수 거래량 오류
연속 장기간 결측(비거래종목 제외) 경고
Step T4. 공시 데이터 확보
지정 기간 공시 조회 (DB 우선)
누락 구간/갭 체크
필요 시 재수집
disclosure_raw → disclosure_parsed
LLM 처리 필요 건을 큐 등록
정책: 중요도 높은 공시 우선
이미 처리된 스키마 버전 있으면 캐시 사용
Step T5. 뉴스 데이터 확보
종목 관련 뉴스 조회 (DB 우선)
기간 누락/희박 구간 검사
필요 시 수집 보완
news_raw → news_parsed
중복 제거/클러스터링
LLM 처리 큐 등록
클러스터 대표 기사 우선 처리
동일 클러스터 멤버는 요약 캐시 재사용 가능
Step T6. SNS/트윗 데이터 확보 (옵션)
계정 기반 + 키워드 기반 조회
종목 연관성 필터링
social_post_raw → social_post_parsed
LLM 처리 큐 등록 (시장 관련성 높은 건 우선)
초기 옵션 전략
quick 모드에서는 SNS 생략 가능
full 모드에서만 포함 가능
Step T7. 기업분석자료/IR 확보 (옵션)
대상 종목 관련 문서 조회
텍스트 추출/정규화
LLM 처리 큐 등록
중기 내러티브 피처 반영
Step T8. 매크로 데이터 머지
사전 적재된 macro_release_actual 조회
기간 내 주요 이벤트 추출
시장/종목 반응 윈도우와 정렬
macro_pressure_score, macro_event_density 등 파생값 계산
Step T9. 피처 계산
가격 기반 기술지표 계산
이벤트 피처 집계
최근 3일/5일/20일 창
텍스트 점수 집계
리스크 점수 집계
feature_store_dailyupsert
Step T10. 신호 생성/평가
룰 게이트 평가
점수 계산
signal_candidate 생성
중복/쿨다운 체크
발송 적합 시 alert_queue enqueue
Step T11. 알림 문구 생성 및 발송 큐 적재
Gemini로 설명용 요약 생성 (필요 시)
채널별 payload 생성 (telegram/sms/kakao)
alert_queue 저장
워커가 발송 후 alert_delivery_log 기록
21.3 Background 파이프라인 상세 (초기 운영 추천 스케줄)
P1 종목 마스터 동기화
주기: 일 1회 (장 종료 후)
처리:
신규상장/상장폐지/종목명변경/상태 변경 반영
산출:
instrument_master, instrument_alias
P2 거래 캘린더 동기화

23. 초기 신호 생성 규칙 초안 (룰 기반 + 점수 기반, 스윙/중기 MVP)
목적: 완전 자동 주문이 아닌 반자동 알림형에 맞는 설명 가능한 초기 전략 룰 정의원칙: 룰 게이트(품질/리스크 방지) + 점수 랭킹(우선순위화)
23.1 신호 분류 체계 (초기)
WATCH_CANDIDATE : 관찰 필요 (이벤트 발생, 확증 부족)
SWING_ENTRY_CANDIDATE : 스윙 진입 후보
SWING_ADD_CANDIDATE : 보유 중 추가검토 후보
RISK_WARNING : 리스크 경고 (악재/거시 악화/이상 급등락)
EXIT_REVIEW_CANDIDATE : 이탈/축소 검토 후보
EVENT_MONITOR : 대형 이벤트 모니터링 알림 (매크로/공시 중심)
23.2 룰 게이트 계층 (선행 필터)
Gate A. 데이터 품질 게이트 (필수)
다음 조건 중 하나라도 불만족 시 신호 생성 억제 또는 confidence 하향:
가격 데이터 최신성 부족
종목 매핑 불확실 (match_confidence 낮음)
핵심 이벤트가 루머성 (rumor_risk_score 높음)
공시/뉴스 텍스트 파싱 실패
LLM 구조화 결과 schema 미검증
결과 처리
rule_gate_passed = false
rule_fail_codes_json 기록
필요 시 WATCH_CANDIDATE로만 제한
Gate B. 유동성/거래가능성 게이트 (초기 권장)
스윙/중기라도 너무 비유동 종목은 알림 품질 낮음.예시 룰(값은 추후 튜닝):
최근 20일 평균 거래대금 하한선
거래정지/관리/투자경고 상태 제외(정책 기반)
급격한 이상변동(단순 테마성) 시 경고 강화
Gate C. 이벤트 신뢰도 게이트
뉴스만 존재 + 공시/공식출처 없음 + 루머리스크 높음 → 진입후보 금지
SNS 단독 신호는 WATCH 또는 EVENT_MONITOR로 제한
공시/공식발표 동반 시 점수 가산 허용
Gate D. 시장 환경 게이트 (매크로/시장 체제)
초기 스윙 전략에서 매우 중요.예:
시장 급락/변동성 급등 체제에서는 진입후보 임계치 상향
매크로 이벤트 직전/직후 혼란 구간은 WATCH 우선
거시 압력 점수 높을 때 고밸류/고변동 종목 알림 보수화
23.3 점수 체계 (초기 버전)
최종 점수는 0~100 스케일 권장.설명 가능성을 위해 factor 별 점수를 분리 저장.
23.3.1 팩터 구성 (초기 예시)
technical_score (0~25)
event_score (0~25)
text_support_score (0~20)
macro_context_score (0~15)
risk_penalty_score (0~20, 감점 항목)
liquidity_score (0~15)
timing_score (0~10) ← 과열/급락/추세 구간 보정
최종 예시:final_score = technical + event + text_support + macro_context + liquidity + timing - risk_penalty
23.3.2 점수 해석 가이드 (초기)
80+ : 고우선순위 검토 후보 (알림 발송 유력)
65~79 : 일반 검토 후보 (조건부 발송)
50~64 : 관찰 후보 (WATCH_CANDIDATE)
<50 : 알림 억제 또는 로그만 저장
(실제 임계값은 초기 운영 후 보정)
23.4 팩터별 세부 규칙 초안
23.4.1 technical_score (스윙/중기 중심)
예시 구성:
중기 추세 정렬 (MA20 > MA60 or price > MA20/60): +0~8
거래량 증가 (평균대비 거래량 급증): +0~6
변동성 안정/확대의 맥락 적합성: +0~4
상대강도(시장 대비): +0~7
주의:
과열 구간(RSI 과도)일 때 가산 축소 또는 timing으로 이동
23.4.2 event_score (뉴스/공시/매크로 이벤트 결합)
예시 구성:
고중요 공시 호재: +0~15
뉴스 이벤트 클러스터 영향도: +0~10
매크로 환경 우호/비우호 보정: -5~+5
이벤트 신선도(최근 1~3일): +0~5
공식성/확정성 높은 이벤트 가산
23.4.3 text_support_score (LLM 구조화 결과 기반)
예시 구성:
뉴스 sentiment + impact_horizon 정합성: +0~8
공시 LLM importance/주주영향도: +0~8
리포트 내러티브 개선: +0~4 (옵션)
주의:
LLM confidence 낮으면 가중치 자동 축소
23.4.4 macro_context_score
예시 구성:
최근 주요지표 서프라이즈가 해당 업종에 우호적: +0~8
시장 risk-on/regime 중립 이상: +0~5
금리/환율 압력 완화: +0~2
거시 악화시:
점수 0 처리 또는 risk_penalty로 이관 가능
23.4.5 risk_penalty_score (감점)
매우 중요. 초기 성능은 감점 규칙이 좌우할 가능성 큼.
예시 감점 항목:
공시 희석 리스크 높음 (유증/CB 등): +0~12 감점
법무/규제 리스크: +0~8 감점
뉴스 루머 리스크: +0~8 감점
과열/단기 급등 후 추격 위험: +0~8 감점
시장 급변동/매크로 이벤트 임박: +0~6 감점
데이터 품질/매핑 신뢰도 낮음: +0~10 감점
23.5 신호 유형 결정 규칙 (초안)
23.5.1 SWING_ENTRY_CANDIDATE
예시 조건 (초기):
rule_gate_passed = true
final_score >= entry_threshold
event_score 또는 technical_score 최소 기준 충족
risk_penalty_score 상한 미만
최근 동일 방향 알림 쿨다운 통과
23.5.2 WATCH_CANDIDATE
예시 조건:
이벤트는 의미 있으나 확증 부족
루머/불확실성 존재
기술적 추세 미완성
매크로 이벤트 임박으로 보류 필요
23.5.3 RISK_WARNING
예시 조건:
악재 공시/법무리스크/희석리스크고점수
루머성 급등/급락 + 거래량 과열
매크로 충격으로 고변동 위험 구간

28.2 뉴스 구조화 프롬프트 템플릿 (news_v1.0.0)
28.2.1 시스템 프롬프트 (초안)
당신의 역할은 주식 자동매매 보조 시스템의 "뉴스 구조화 엔진"입니다.목표:- 뉴스 기사(제목/본문)를 투자판단용 정형 데이터(JSON)로 변환합니다.- 최종 매수/매도 판단을 하지 않습니다.- 원문에 없는 사실을 생성하지 않습니다.중요 규칙:1) 출력은 반드시 JSON 객체 하나만 반환합니다.2) 제공된 스키마 필드만 사용합니다. 모르는 값은 null 또는 "unclear"로 처리합니다.3) 추정/불확실한 내용은 uncertainty_flags에 명시합니다.4) 숫자/날짜/단위는 가능한 한 원문 표현을 유지합니다.5) 기사 문체(과장/추정/루머)를 구분하여 rumor_risk_score와 event_stage에 반영합니다.6) 투자 자문(매수/매도/익절/손절 지시)을 하지 않습니다.평가 관점:- event_type 분류 정확성- 종목/섹터/시장 영향범위 추정의 일관성- 사실과 해석의 분리- 신뢰도/루머 위험도 보수적 평가
28.2.2 사용자 프롬프트 템플릿 (초안)
[task_meta]task_type: news_structuringschema_version: 1.0.0language: ko[input_context]as_of_time_utc: {{as_of_time_utc}}source_name: {{source_name}}publisher_name: {{publisher_name}}publish_time_utc: {{publish_time_utc}}candidate_instruments: {{candidate_instruments_json}}news_cluster_context: {{news_cluster_context_json}}  # 없으면 nullrecent_price_context: {{recent_price_context_json}}  # 없으면 null[source_text]title: {{title}}body: {{body_text}}[constraints]- JSON only- no markdown- no code fence- if uncertain, use uncertainty_flags- if candidate_instruments 정보가 불충분하면 target_instruments를 빈 배열로 둘 수 있음[output_schema_fields]{  "schema_version": "1.0.0",  "task_type": "news_structuring",  "event_type": "earnings|guidance|order|regulation|legal|management|macro_linked|market_commentary|rumor|other",  "event_stage": "rumor|announced|confirmed|followup|analysis",  "target_scope": "single_stock|sector|market|macro",  "target_instruments": [{"name":"","ticker":"","relevance_score":0.0}],  "sentiment_score": -1.0 to 1.0,  "impact_direction": "positive|negative|mixed|neutral|unclear",  "impact_horizon": "intraday|1d|3d|1w|1m|longer|unclear",  "novelty_score": 0.0 to 1.0,  "reliability_score": 0.0 to 1.0,  "rumor_risk_score": 0.0 to 1.0,  "numerical_facts": [{"label":"","value_text":"","unit":""}],  "key_points": ["..."],  "bull_points": ["..."],  "bear_points": ["..."],  "uncertainty_flags": ["..."],  "confidence_score": 0.0 to 1.0,  "reason_codes": ["..."],  "summary_short": "...",  "summary_bullets": ["..."]}
28.2.3 뉴스 구조화 후 서버측 검증 규칙 (권장)

28.3.1 시스템 프롬프트 (초안)
당신의 역할은 주식 자동매매 보조 시스템의 "공시 구조화 엔진"입니다.목표:- 공시 제목/본문/첨부 추출 텍스트를 주주가치 영향과 리스크 관점의 정형 JSON으로 변환합니다.- 사실관계 구조화를 우선하고, 시장반응 예측은 보수적으로 수행합니다.중요 규칙:1) 출력은 반드시 JSON 객체 하나만 반환합니다.2) 원문에 없는 숫자/사실을 만들지 않습니다.3) 단위(억원, 백만원, %, 주, 기간)를 엄격히 보존합니다.4) 확정되지 않은 해석은 uncertainty_flags에 표시합니다.5) 최종 매수/매도 지시를 하지 않습니다.6) 정정공시 여부/변경 포인트가 보이면 must_check_items에 포함합니다.평가 관점:- 공시 유형 분류 정확성- 숫자/단위 추출 정확성- 희석/재무/법무/지배구조 리스크의 보수적 판단
