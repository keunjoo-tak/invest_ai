* 현재 서비스는 **Market Regime → Stock Decision → Action Planner**의 계층형 구조로 운영된다.
* 리서치 리포트는 **시장/섹터/종목 판단의 근거 레이어**로 사용하고, 가격·수급·재무·뉴스를 대체하지 않는다.
* 수집 대상은 **공개 공개형(public open)**, **로그인/고객권한형(authenticated)**, **상용 라이선스형(licensed)** 으로 분리한다.
* 원문 저작권과 재배포 제한이 크므로, **사용자 노출은 원문 전문이 아니라 파생 데이터·근거 스니펫·출처 링크 중심**으로 설계한다. 삼성증권 리포트는 무단 복제·배포·전송·변경·대여를 금지한다고 명시하고 있고, Morgan Stanley는 별도 Research Portal 접근을 요구하며, Nomura는 자료가 전문투자자/기관투자자 대상이라고 밝히고 있다. Morningstar도 별도 Licensed Research 상품을 운영한다. ([Samsung Pop][1])

## 2) 수집 정책 설계

### 2-A. 권한 모델

모든 소스에 아래 속성을 강제한다.

* `access_tier`: `PUBLIC_OPEN | AUTHENTICATED | LICENSED | MANUAL`
* `redistribution_policy`: `DERIVED_ONLY | INTERNAL_TEXT_ONLY | LICENSED_FULLTEXT`
* `source_of_truth`: `OFFICIAL_SITE | LICENSED_VENDOR`
* `entitlement_required`: `true/false`

핵심 규칙은 다음이다.

1. `PUBLIC_OPEN` 은 원문 내부 저장은 가능하되, 사용자 노출은 **짧은 스니펫 + 출처 + 파생 피처** 중심.
2. `AUTHENTICATED` 와 `LICENSED` 는 **조직이 합법적 권한을 가진 경우에만** 세션/토큰 기반 수집.
3. `LICENSED_FULLTEXT` 가 아닌 경우, LLM 출력은 **원문 재생산 금지**, 파생 인사이트만 제공.

### 2-B. 수집 어댑터 타입

소스별 하드코딩이 아니라 **어댑터 + 프로파일** 구조로 구현해야 한다.

* `HtmlListPdfAdapter`
  HTML 목록 페이지 → 상세/다운로드 링크 추출 → PDF 저장
  적용: 삼성, 미래에셋, NH, 신한, 키움, 메리츠 일부

* `HtmlSearchAdapter`
  검색 결과 페이지 기반 크롤링
  적용: 한국투자, 일부 KB/유진

* `HtmlArticleAdapter`
  HTML 기사/인사이트 본문 추출
  적용: UBS, BlackRock, Goldman, Fidelity, Schroders, PIMCO

* `AuthenticatedPortalAdapter`
  로그인 세션/쿠키 기반 목록·첨부 수집
  적용: 하나증권, Morgan Stanley Portal 등

* `LicensedFeedAdapter`
  벤더 API/export/SFTP 기반 적재
  적용: Morningstar, AlphaSense, S&P Capital IQ Pro

* `ManualDropAdapter`
  이메일 첨부, SFTP, 운영자 업로드
  적용: 계약형 배포 PDF, 폐쇄형 기관자료

## 3) 전체 파이프라인 아키텍처

### 3-A. 권장 스택

* Language: `Python 3.12`
* Orchestration: `Prefect`
* HTTP/Browser: `httpx + BeautifulSoup + Playwright(fallback)`
* Parsing: `PyMuPDF`, `pdfplumber`, `python-docx`, `mammoth`, `trafilatura`
* Queue/Cache: `Redis`
* Metadata DB: `PostgreSQL`
* Vector/Search: `pgvector` 로 시작, 필요 시 `OpenSearch` 추가
* Blob Storage: `local`
* API/Admin: `FastAPI`

### 3-B. 단계

1. `source_discovery_job`
   목록 페이지/검색 페이지/벤더 인덱스에서 신규 문서 URL 탐지

2. `document_fetch_job`
   원본 파일/HTML 저장, content hash 생성, 버전 관리

3. `document_parse_job`
   MIME 기반 파싱, 본문/표/메타데이터/첨부 분리

4. `document_normalize_job`
   문서 유형 분류, 엔티티 매핑, 카테고리 표준화

5. `feature_extraction_job`
   리포트 수준 피처, 종목/산업/시장 시그널 생성

6. `service_projection_job`
   Market Regime / Stock Decision / Action Planner 용 materialized snapshot 생성

7. `qa_monitoring_job`
   실패 문서, 파싱 품질, 중복, 결측, entitlement 오류 감시

## 4) 적재 구조(Storage Design)

### 4-A. 저장 원칙

* **원본은 blob storage**
* **문서 메타/피처/서비스 스냅샷은 PostgreSQL**
* **검색과 RAG는 pgvector + tsvector로 시작**
* **고빈도 시계열 피처가 늘면 ClickHouse/OpenSearch를 추가**

## 5) 파싱 및 전처리 설계

### 5-A. MIME별 처리

* `PDF`

  * 1차: `PyMuPDF` 로 텍스트/페이지 메타 추출
  * 2차: `pdfplumber` 로 표 추출
  * 3차: 스캔 PDF인 경우에만 OCR fallback
  * 규칙: 첫 1~3페이지에서 제목/애널리스트/투자의견/목표주가/발간일 집중 추출

* `DOCX`

  * `python-docx` + `mammoth`
  * 본문, heading, 표, 각주 추출
  * embedded object는 별도 아티팩트 저장

* `HTML`

  * `trafilatura` → 실패 시 `BeautifulSoup` 룰 기반
  * 본문, 요약, 작성자, 발행일, 카테고리, 다운로드 첨부 추출

## 7) 서비스 적용 방식

### 7-A. Market Regime

출력 계약:

```json
{
  "regime_label": "Risk-On / Neutral / Risk-Off",
  "regime_score": 0.0,
  "supporting_factors": [],
  "opposing_factors": [],
  "sector_strength": [],
  "regional_preferences": [],
  "strategy_hints": [],
  "invalidation_conditions": [],
  "data_confidence": 0.0,
  "evidence_docs": []
}
```

적용 방식:

* daily/weekly 전략 리포트와 은행/글로벌 macro outlook를 집계
* 하우스별 stance 를 recency-weighted average 로 통합
* sector_overweight/underweight 언급을 섹터 강약도로 투영
* 사용자에게는 **“현재 체제 / 근거 / 반증 / 전략 힌트 / 신뢰도”** 순서로 제공

### 7-B. Stock Decision

출력 계약:

```json
{
  "stock_id": "KRX:005930",
  "decision_label": "Buy / Hold / Reduce / Avoid",
  "horizon_scores": {
    "short": 0.0,
    "swing": 0.0,
    "midterm": 0.0
  },
  "supporting_drivers": [],
  "opposing_drivers": [],
  "research_consensus": {
    "recommendation_score": 0.0,
    "dispersion_score": 0.0,
    "target_price_upside_pct": 0.0
  },
  "catalysts": [],
  "risks": [],
  "checkpoints": [],
  "data_confidence": 0.0,
  "evidence_docs": []
}
```

적용 방식:

* 기업분석 + 산업분석 + 전략 리포트 + 현재 Market Regime 를 결합
* 동일 종목에 대한 **최근 30/90일 투자의견 변화, 목표가 변화, 실적 추정 변경**을 누적
* 사용자에게는 **“결론 / 찬성 근거 / 반대 근거 / 근접 촉매 / 체크포인트”** 순으로 제공

### 7-C. Action Planner

출력 계약:

```json
{
  "action_label": "Immediate / Staged / Wait / Reduce",
  "base_scenario": {},
  "bull_scenario": {},
  "bear_scenario": {},
  "entry_rules": [],
  "add_rules": [],
  "exit_rules": [],
  "invalidators": [],
  "review_triggers": [],
  "why_now": [],
  "why_not_now": []
}
```

적용 방식:

* Stock Decision 결과를 그대로 반복하지 않고,
* `regime_alignment + event_risk + conviction + volatility` 를 다시 평가해서
* **실행 시나리오**로 변환
* 사용자에게는 **“지금 할 행동 / 하지 말아야 할 행동 / 다음 점검 조건”** 을 명확히 제시

## 12) 최종 구현 원칙

1. **소스별 크롤러를 하드코딩하지 말고, adapter + profile + selector config 구조로 구현할 것.**
2. **원문 저장과 사용자 노출을 분리할 것.**
3. **리서치 리포트는 독립 진실값이 아니라 해석/설명 강화 레이어로 사용할 것.**
4. **기업분석은 단일 하우스가 아니라 cross-house consensus 와 dispersion 을 함께 볼 것.**
5. **Market Regime는 은행/글로벌 매크로 리포트 비중을 높이고, Stock Decision은 기업/산업 리포트 비중을 높일 것.**
6. **Action Planner는 점수 요약이 아니라 실행 규칙 산출기로 만들 것.**
7. **글로벌 sell-side 기업분석은 공개 웹만으로는 불완전하므로, licensed lane 을 전제로 설계할 것.** UBS/BlackRock/Goldman/JPM/Fidelity/Schroders/PIMCO의 공개 영역은 주로 시장·경제·산업 인사이트 중심이고, Morningstar/Morgan Stanley/Nomura/AlphaSense/S&P는 별도 라이선스·포털·권한 모델이 존재한다. ([Japan][19])
