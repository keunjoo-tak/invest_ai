첫번째 답변
====================

실무 기준으로는 먼저 **예측 대상**을 고정하는 편이 좋습니다. 보통은 “절대 주가”보다 **다음날/다음주 수익률**, **시장 대비 초과수익률**, **상승/하락 방향**을 타깃으로 둡니다.

아래는 **한국 주식의 특정 종목 주가 예측**에 많이 쓰는 변수와, **각 데이터를 수집할 수 있는 위치**를 묶어서 정리한 목록입니다.

1. **시세·유동성 변수**  
시가/고가/저가/종가, 거래량, 거래대금, 시가총액, 회전율, 갭수익률, 1·5·20일 수익률, 이동평균 괴리율, 변동성(rolling std, ATR), 거래량 급증 여부 같은 기본 시계열입니다.  
수집 위치는 **KRX Data Marketplace**의 전종목/개별종목 시세·기본정보·시세추이와, **KRX Open API**의 유가증권·코스닥·코넥스 일별매매정보/종목기본정보가 기본입니다. ([data.krx.co.kr](https://data.krx.co.kr/contents/MDC/COMS/client/MDCCOMS002_S3.cmd?utm_source=chatgpt.com))

2. **수급 변수**  
외국인·기관·개인 순매수(수량/대금), 투자자별 거래실적, 프로그램매매, 협의대량거래 추이입니다. 한국 시장에서는 종목 자체보다 **누가 샀는지**가 단기 방향성에 꽤 중요하게 작동하는 경우가 많습니다.  
수집 위치는 **KRX Data Marketplace**의 투자자별 거래실적(개별종목), 프로그램매매, 협의대량거래실적 추이입니다. ([data.krx.co.kr](https://data.krx.co.kr/contents/MDC/MDI/outerLoader/index.cmd?kosdaqGlobalYn=1&locale=ko_KR&screenId=MDCSTAT023&utm_source=chatgpt.com))

3. **공매도·대차 변수**  
공매도 거래대금/거래비중, 공매도 순보유잔고, 공매도 과열종목 여부, 대차거래 추이, 대차잔고, 내·외국인 대차잔고 비교입니다.  
수집 위치는 **KRX Data Marketplace**의 공매도 통계·대차 정보, 그리고 **한국예탁결제원 Seibro**의 주식대차·대차종목순위·내외국인대차잔고비교 화면입니다. ([data.krx.co.kr](https://data.krx.co.kr/contents/MDC/MAIN/main/index.cmd?utm_source=chatgpt.com))

4. **시장·업종·상대강도 변수**  
KOSPI/KOSDAQ/업종지수 수익률, 종목의 시장 대비 초과수익률, 업종 대비 상대강도, 지수 편입 여부, ETF/ETN/ELW 관련 흐름입니다. 개별종목 예측도 결국 시장/업종 레짐 영향을 크게 받습니다.  
수집 위치는 **KRX Data Marketplace**의 지수 시세·지수구성종목·PER/PBR/배당수익률, 그리고 **KRX Open API**의 지수·ETF/ETN/ELW 일별매매정보입니다. ([data.krx.co.kr](https://data.krx.co.kr/contents/MDC/COMS/client/MDCCOMS002_S3.cmd?utm_source=chatgpt.com))

5. **파생상품 변수**  
KOSPI200 선물/옵션 수익률·거래량, 개별주식 선물/옵션 거래정보, 현선물 동조화 지표 같은 변수입니다. 대형주나 지수 민감 종목에서는 특히 검토할 가치가 있습니다.  
수집 위치는 **KRX Open API**의 파생상품 서비스로, 선물·옵션·주식선물·주식옵션 일별매매정보를 제공합니다. ([openapi.krx.co.kr](https://openapi.krx.co.kr/contents/OPP/INFO/service/OPPINFO004.cmd?utm_source=chatgpt.com))

6. **정태 팩터·밸류에이션 변수**  
PER, PBR, 배당수익률, 시가총액, 상장시장(KOSPI/KOSDAQ), 외국인보유량, 업종분류, 발행주식수/주식발행내역입니다.  
수집 위치는 **KRX Data Marketplace**의 PER/PBR/배당수익률, 외국인보유량, 업종분류/상장회사 상세검색과 **KIND**의 상장법인목록·주식발행내역입니다. ([data.krx.co.kr](https://data.krx.co.kr/contents/MDC/COMS/board/MDCCOMS010_S1.cmd?boardId=MDCINFO004&utm_source=chatgpt.com))

7. **재무·실적 변수**  
매출, 영업이익, 순이익, EPS/BPS, 영업이익률, 부채비율, 유동비율, 현금흐름, CAPEX, R&D, 제품/사업부 매출 비중, 주석 정보 등입니다. 중기 예측에서는 가격 변수보다 이 쪽이 더 중요해질 수 있습니다.  
수집 위치는 **OpenDART**의 정기보고서 주요정보, 재무정보, 주요계정 비교, 주석 조회, XBRL 재무제표 원문/주석 다운로드가 핵심입니다. 보조적으로 **KIND**의 회사재무비교도 활용 가능합니다. ([opendart.fss.or.kr](https://opendart.fss.or.kr/guide/detail.do?apiGrpCd=DS001&apiId=2019001&utm_source=chatgpt.com))

8. **공시 이벤트 변수**  
실적발표일, 공급계약, 시설투자, 합병/분할, 유상증자, 전환사채/신주인수권부사채, 소송, 영업정지, 관리종목/상장폐지 위험 같은 이벤트 더미입니다.  
수집 위치는 **OpenDART**의 공시검색·주요사항보고서 주요정보와 **KIND**의 오늘의공시/회사별검색/상장폐지현황/투자유의사항입니다. ([opendart.fss.or.kr](https://opendart.fss.or.kr/guide/detail.do?apiGrpCd=DS001&apiId=2019001&utm_source=chatgpt.com))

9. **지분·거버넌스·주주환원 변수**  
5% 대량보유 보고, 임원·주요주주 소유/거래계획, 자기주식 취득·처분, 배당, IR 일정/IR 자료, 기업지배구조 보고서, ESG 등급 같은 변수입니다.  
수집 위치는 **OpenDART**의 지분공시 종합정보(D001, D002, D005)와 **KIND**의 자사주취득/처분, 배당정보, IR 일정/자료실, 기업지배구조/ESG 메뉴입니다. ([opendart.fss.or.kr](https://opendart.fss.or.kr/guide/detail.do?apiGrpCd=DS001&apiId=2019001&utm_source=chatgpt.com))

10. **거시금융 변수**  
기준금리, CD(91일), 국고채 3년, 회사채 3년 AA-, 신용스프레드, 환율, 통화량, CPI, 생산, 고용 등입니다. 종목 단독 모델이라도 시장 레짐 변수로 넣는 편이 안정적입니다.  
수집 위치는 **한국은행 ECOS Open API**와 **KOSIS OpenAPI**입니다. ECOS 일일지표 화면에는 CD(91일), 국고채(3년), 회사채(3년, AA-) 같은 지표가 제공됩니다. ([ecos.bok.or.kr](https://ecos.bok.or.kr/api/?utm_source=chatgpt.com))

11. **실물·무역·산업 선행지표 변수**  
품목별/국가별 수출입, HS코드 기준 물동량, 산업생산, 소재·부품·장비 동향 같은 변수입니다. 반도체, 자동차, 화학, 해운처럼 수출 민감 업종에서는 효과가 큽니다.  
수집 위치는 **관세청 수출입무역통계/관세청 빅데이터포털**, **data.go.kr의 관세청 국가별 수출입 OpenAPI**, 그리고 **KOSIS**입니다. ([bigdata.customs.go.kr](https://bigdata.customs.go.kr/web/cten/2/ctenView.do?utm_source=chatgpt.com))

12. **뉴스·텍스트·검색관심도 변수**  
뉴스 건수, 제목/본문 감성, 키워드 빈도, 개체명, 공시 원문 임베딩, 검색량 트렌드입니다. 이벤트 드리븐 종목이나 테마주에서는 유용합니다.  
수집 위치는 **네이버 Search API(뉴스 검색)**, **네이버 DataLab 검색어 트렌드 API**, **BIG KINDS의 뉴스검색·분석/OPEN API/기업 뉴스 화면**입니다. 애널리스트 기대치까지 넣고 싶으면 **FnGuide Company Guide**의 컨센서스/리포트 요약 같은 상용 DB를 추가하는 경우가 많습니다. ([developers.naver.com](https://developers.naver.com/docs/serviceapi/search/news/news.md?utm_source=chatgpt.com))

13. **초단기 모델용 실시간 데이터**  
장중 예측을 하려면 일별 데이터만으로는 부족합니다. 이 경우에는 별도 실시간 데이터 라인을 검토해야 합니다.  
수집 위치는 **KRX OPEN API의 데이터 분배상품**입니다. KRX는 실시간/종가/참조정보 분배 상품을 전용망·인터넷망·FTP 방식으로 안내하고 있습니다. ([openapi.krx.co.kr](https://openapi.krx.co.kr/contents/OPP/DATA/OPPDATA002.jsp?utm_source=chatgpt.com))

처음 모델을 만든다면, 변수는 일단 아래 정도로 시작하는 것이 보통 효율적입니다.

실무상 주의할 점은 세 가지입니다.  
첫째, **공시·재무·거시 변수는 반드시 “발표일 기준”으로 정렬**해야 합니다.  
둘째, **절대 주가보다 수익률/방향성 예측이 보통 더 안정적**입니다.  
셋째, **한국 소형주는 거래정지·유상증자·관리종목·테마 급등락 영향이 커서 이벤트 더미를 꼭 분리**하는 편이 좋습니다.

원하시면 다음 단계로 바로 이어서, **“한국 주식 주가예측용 데이터 스키마(테이블 설계)”** 또는 **“Python 수집 파이프라인 구조”**까지 설계해드리겠습니다.




두번째 답변
====================

가능합니다. 아래 구조는 **한국 개별 종목 예측 파이프라인**에 **글로벌 매크로·이슈 수집 계층**을 붙인 Python 기준 설계입니다. 핵심은 `시장 데이터`, `공시/재무`, `거시지표`, `이슈·이벤트`를 모두 **동일한 시간축과 가용시점(available time)** 기준으로 정렬하는 것입니다. 국내 원천은 KRX Open API/Data Marketplace, OpenDART, ECOS, KIND 계열로 두고, 해외는 미국(FRED/ALFRED, BLS, BEA, Treasury), 유럽(ECB, Eurostat, OECD), 중국(NBS, PBOC, 중국 해관), 글로벌(IMF, OECD, World Bank optional)로 묶는 구성이 안정적입니다. ([openapi.krx.co.kr](https://openapi.krx.co.kr/contents/OPP/INFO/service/OPPINFO004.cmd?utm_source=chatgpt.com))

가장 중요한 설계 원칙은 두 가지입니다. 첫째, **look-ahead leakage 방지**를 위해 모든 데이터에 `observation_date`, `release_at`, `available_at`, `ingested_at`, `vintage/revision`를 분리 저장해야 합니다. 둘째, 글로벌 이벤트는 **캘린더 기반**으로 수집해야 합니다. FRED는 ALFRED를 통해 과거 시점에 실제 공개되었던 vintage를 조회할 수 있고, Fed·ECB·NBS·Eurostat는 공식 일정/릴리즈 캘린더를 제공합니다. 모델 학습용 데이터셋은 반드시 “그 날짜에 실제로 알 수 있었던 정보만” 재구성해야 합니다. ([alfred.stlouisfed.org](https://alfred.stlouisfed.org/?utm_source=chatgpt.com))

전체 구조는 아래처럼 두면 됩니다.

```text

        ↓

        ↓

[Curated Warehouse]
  ├─ fact_market_*

```

실무적으로는 **오브젝트 스토리지 + Parquet**를 원천/정규화 저장소로 쓰고, **PostgreSQL**을 메타데이터·서빙용 웨어하우스로 두는 구성이 무난합니다. 로컬 개발과 리서치에는 DuckDB를 같이 붙이면 빠릅니다. 오케스트레이션은 Airflow나 Prefect 중 하나로 두고, 수집기는 모두 idempotent 하게 만듭니다.

국내 블록은 아래처럼 나누면 됩니다.  
`KRX`는 일별 주식·지수·ETF/ETN·파생상품 정보를, `OpenDART`는 공시 원문과 주요 공시/재무 정보를, `ECOS`는 한국 거시지표를 담당하게 두는 구조입니다. `KIND`는 자사주, IR, 배당, 거래소 공시 보강용으로 붙입니다. ([openapi.krx.co.kr](https://openapi.krx.co.kr/contents/OPP/INFO/service/OPPINFO004.cmd?utm_source=chatgpt.com))

해외 거시지표 블록은 아래처럼 설계하면 됩니다.

- **미국**: `FRED/ALFRED`를 코어 허브로 쓰고, 고빈도·공식 발표 계열은 `BLS`, `BEA`, `Treasury`를 직접 붙입니다. FRED API는 경제시계열 조회를 지원하고 ALFRED는 vintage/real-time period를 보존합니다. BLS는 Public API v2 시그니처를 공개하고, BEA는 JSON/XML 기반 API를 제공합니다. Treasury는 금리/수익률 데이터 공개 엔드포인트를 제공합니다. ([fred.stlouisfed.org](https://fred.stlouisfed.org/docs/api/fred/))  
- **유럽**: `ECB Data Portal`과 `Eurostat`를 기본으로 두고, cross-country 비교는 `OECD`를 보강용으로 씁니다. ECB는 SDMX 2.1 REST, Eurostat는 STATISTICS·SDMX 3.0·SDMX 2.1·CATALOGUE API를 제공합니다. OECD도 SDMX 기반 API를 제공합니다. ([data.ecb.europa.eu](https://data.ecb.europa.eu/help/api/overview))  
- **중국**: `NBS National Data`, `PBOC`, `China Customs`를 각각 실물지표·통화지표·무역지표로 나눕니다. NBS 영어 사이트는 Statistical Database, 월/분기/연간 데이터, RSS, Release Calendar를 제공하고, PBOC는 금융통계 보고서와 공개시장조작·통화관련 발표 페이지를 운영하며, 중국 해관은 수출입 통계를 공개합니다. 중국은 미국·유럽보다 API 일관성이 떨어져 **캘린더 기반 HTML 수집기**를 별도로 두는 편이 안정적입니다. 이 마지막 판단은 공식 사이트 구조를 보고 한 설계상 추론입니다. ([stats.gov.cn](https://www.stats.gov.cn/english/))  
- **글로벌 공통**: IMF와 OECD를 공통 매크로 허브로 두고, World Bank Indicators API v2를 장기·국가 비교용 백필 소스로 붙일 수 있습니다. IMF는 SDMX 2.1/3.0 API를, World Bank는 v2 Indicators API를 제공합니다. ([data.imf.org](https://data.imf.org/en/Resource-Pages/IMF-API))

글로벌 이슈 수집은 **정형 거시지표와 분리**해서 설계해야 합니다. 저는 `official_event_stream`과 `broad_issue_stream` 두 개로 나누는 편을 권합니다. Fed와 ECB는 공식 RSS를 제공하고, Fed는 FOMC 캘린더와 자료 공개 일정을, ECB는 Governing Council 일정과 기자회견 일정을 제공합니다. NBS 영어 사이트는 RSS와 Release Calendar를 제공하고, Eurostat도 공식 Release Calendar를 운영합니다. 그래서 “정책회의”, “통계 발표”, “예고된 이벤트”는 뉴스보다 먼저 **캘린더 테이블**에 등록한 뒤, 릴리즈 시각 전후로 poll 하는 구조가 좋습니다. ([federalreserve.gov](https://www.federalreserve.gov/feeds/feeds.htm))

`broad_issue_stream`은 GDELT와 공식 기관 보도자료를 섞으면 됩니다. GDELT는 100개 이상 언어의 글로벌 뉴스와 이벤트/테마/감정 신호를 근실시간으로 제공하고 15분 단위 업데이트 구조를 갖습니다. 여기에 Fed/ECB/NBS/PBOC/OECD/World Bank 같은 공식 발표 페이지를 별도 수집해 “정책성 이슈”의 신뢰도를 높입니다. 검색 관심도는 이제 **공식 Google Trends API가 alpha**로 존재하지만, 현재는 제한된 alpha tester만 접근할 수 있으므로 이 커넥터는 feature flag 뒤에 두는 것이 맞습니다. Google 문서상 API는 최근 5년 데이터, 일/주/월/년 집계, 국가·하위지역 비교, consistently scaled data를 지원합니다. ([gdeltproject.org](https://www.gdeltproject.org/))

수집기 인터페이스는 통일해 두는 게 좋습니다.

```python
from dataclasses import dataclass
from typing import Any, Iterable
from abc import ABC, abstractmethod
from datetime import datetime, date

@dataclass
class CollectJob:
    source: str
    dataset: str
    start: date | None = None
    end: date | None = None
    params: dict[str, Any] | None = None

@dataclass
class RawRecord:
    source: str
    dataset: str
    fetched_at: datetime
    payload: dict[str, Any] | str
    payload_hash: str

@dataclass
class NormalizedRecord:
    source: str
    dataset: str
    obs_date: date | None
    release_at: datetime | None
    available_at: datetime | None
    ingested_at: datetime
    entity_id: str | None
    values: dict[str, Any]
    raw_hash: str

class BaseCollector(ABC):
    @abstractmethod
    def fetch(self, job: CollectJob) -> Iterable[RawRecord]:
        ...

class BaseNormalizer(ABC):
    @abstractmethod
    def normalize(self, raw: Iterable[RawRecord]) -> Iterable[NormalizedRecord]:
        ...
```

핵심 구현 포인트는 다섯 개입니다.

1. **시간대 표준화**  
모든 원천은 UTC로 저장하고, 별도 `source_tz`를 둡니다. 한국(KST), 미국(ET), 유럽(CET/CEST), 중국(CST)을 혼용하면 릴리즈 시점 오류가 쉽게 납니다.

2. **캘린더 우선 수집**  
FOMC, ECB, Eurostat, NBS처럼 일정이 공개되는 소스는 캘린더를 먼저 수집하고, 해당 시각에만 공격적으로 polling 합니다. 이렇게 해야 rate limit과 결측을 줄일 수 있습니다. ([federalreserve.gov](https://www.federalreserve.gov/monetarypolicy/fomccalendars.htm))

3. **원천 원문 보존**  
JSON/XML/HTML/RSS 원문은 절대 버리지 말고 `raw lake`에 저장합니다. 중국계 소스처럼 HTML 위주인 경우 파서가 바뀌어도 재처리가 가능해야 합니다. NBS/PBOC/China Customs는 이 방식이 특히 중요합니다. 이건 공식 페이지 구조를 기준으로 한 설계상 판단입니다. ([stats.gov.cn](https://www.stats.gov.cn/english/))

4. **지표 메타카탈로그 운영**  
`series_catalog`에 `source_series_code`, `freq`, `unit`, `geo`, `release_lag_rule`, `transform_rule`, `is_revision_prone`를 넣어 둡니다. 파이프라인 코드보다 카탈로그 변경으로 운영 범위를 넓히는 구조가 훨씬 관리하기 쉽습니다.

5. **특징 생성 분리**  
수집 파이프라인과 feature 파이프라인은 분리합니다. 수집은 “사실 저장”, feature는 “모델용 변환”입니다. 그래야 모델이 바뀌어도 원천 적재를 재사용할 수 있습니다.

마지막으로, 이 설계에서 바로 시작할 **MVP 범위**는 아래 정도가 적절합니다.

- 국내: KRX 가격/수급/공매도 + DART 공시/재무 + ECOS
- 미국: FRED/ALFRED + BLS + BEA + Treasury
- 유럽: ECB + Eurostat
- 중국: NBS + PBOC + China Customs
- 글로벌 이슈: Fed/ECB/NBS/Eurostat 캘린더 + GDELT + 공식 RSS/보도자료
- 출력: `feature_symbol_daily`, `training_snapshot`

이렇게 시작하면 한국 종목별 예측에서 바로 쓰는 핵심 블록은 대부분 커버됩니다.
